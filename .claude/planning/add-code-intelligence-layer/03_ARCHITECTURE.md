## Component Design

### Component 1: Symbol Index

**Responsibility:** Provide a structured, queryable list of top-level code symbols with file/line references.

**Location:** Generated by `repo-map.md`, stored in `01_DISCOVERY.md ## Symbol Index`

**Input:** Source files identified by Glob + language detection from `/discover`

**Output format (compact):**
```
## Symbol Index

func:calculateTotal:src/billing/calculator.ts:15
class:PaymentService:src/billing/service.ts:8
iface:PaymentConfig:src/billing/types.ts:3
func:validateCard:src/billing/validator.ts:22
export:DEFAULT_CURRENCY:src/billing/constants.ts:1
```

**Format rules:**
- One symbol per line: `{type}:{name}:{file}:{line}`
- Types: `func`, `class`, `iface`, `type`, `const`, `export`, `enum`, `trait`, `struct`
- Sorted by file path, then line number
- Budget: ≤1K tokens (~4K chars, ~80 symbols max)

**Progressive truncation (same tiers as repo map):**

| Repo size | Symbol index scope |
|-----------|-------------------|
| < 100 files | All source files |
| 100–200 files | Primary source dirs only (`src/`, `lib/`, `app/`, `pkg/`) |
| 200–500 files | Top 50 files by directory priority |
| > 500 files | Top 30 files from primary source dirs + note: "Partial index" |

### Component 2: Dependency Graph

**Responsibility:** Map module-level import/export relationships for candidate files.

**Location:** Built in-context during `researching-code` Step 0b. NOT persisted to file.

**Input:** Candidate files from Step 0a (repo map + symbol index analysis)

**Procedure:**
1. For each candidate file (5-15 files), run Grep with the language-appropriate import pattern
2. Resolve import paths to actual files in the repo (relative → absolute)
3. Identify corresponding test files via naming convention (`*.test.*`, `*.spec.*`, `test_*`, `*_test.*`)
4. Build a mental adjacency list: `file → [imports..., imported-by..., tested-by...]`

**Import patterns per language:**

| Language | Grep Pattern | Example Match |
|----------|-------------|---------------|
| JS/TS | `^import .+ from ['"](.+)['"]` | `import { foo } from './utils'` |
| JS/TS (require) | `require\(['"](.+)['"]\)` | `const x = require('./utils')` |
| Python | `^(import \|from )(\S+)` | `from utils.helpers import calc` |
| Go | `"([^"]+)"` (inside import block) | `"github.com/pkg/errors"` |
| PHP | `^use (.+);` | `use App\Services\PaymentService;` |
| Rust | `^use (.+);` | `use crate::billing::calculator;` |
| Generic | `^import\|^require\|^use\|^from .+ import` | Catches most patterns |

**Test file detection patterns:**
- `{name}.test.{ext}`, `{name}.spec.{ext}` (JS/TS)
- `test_{name}.{ext}`, `{name}_test.{ext}` (Python, Go)
- `tests/{name}.{ext}`, `__tests__/{name}.{ext}` (directory-based)

### Component 3: Reranker

**Responsibility:** Score and re-order Level 2 retrieval results by relevance to the current task.

**Location:** Step 0d in `researching-code` skill.

**Activation condition:** Only runs when Level 2 returns > 5 results. For ≤ 5, use all results without reranking.

**Scoring factors (3-factor heuristic):**

| Factor | Weight | How to Evaluate |
|--------|--------|-----------------|
| **Keyword overlap** | 40% | Count how many task-description keywords appear in the file path + symbol names. Normalize by total keywords. |
| **Dependency proximity** | 35% | Score 3 if directly imported by a seed file, 2 if it imports a seed file, 1 if shares a dependency with seed, 0 otherwise. |
| **File-type priority** | 25% | source=3, test=2, config=1, doc/generated=0 |

**Procedure:**
1. For each candidate file from Level 2 results, mentally compute the 3 scores
2. Composite score = (keyword × 0.4) + (proximity × 0.35) + (filetype × 0.25)
3. Re-order by composite score descending
4. Take top-N (default N=8)

**This is a mental procedure, not code.** The LLM evaluates these factors as part of its reasoning. The prompt provides the scoring rubric; the LLM applies it.

### Component 4: Context Pack Builder

**Responsibility:** Assemble the final set of files to read for deep analysis in Steps 1-2.

**Location:** Step 0e in `researching-code` skill.

**Input:** Top-N reranked files + dependency graph from Step 0b

**Procedure:**
1. Start with top-N reranked seed files (default N=5)
2. For each seed file, check the dependency graph:
   - Add up to 2 direct imports that aren't already in the pack
   - Add the corresponding test file if it exists
3. Apply hard cap: **≤ 8 files total** in the context pack
4. If over budget, drop test files first, then lowest-scored imports
5. Read these files (or key sections for large files)

**Output:** The list of files that Steps 1-2 should focus on. This replaces the ad-hoc "find 5-15 files" approach with a structured assembly.

**Token budget guidance:**
- Small files (< 100 lines): read entirely
- Medium files (100-300 lines): read imports + exported symbols + relevant functions
- Large files (> 300 lines): read only the sections matching task keywords

### Component 5: Session Persistence

**Responsibility:** Avoid redundant computation across phases and sessions.

**Design: No new infrastructure. Two existing mechanisms suffice.**

| Persistence Scope | Mechanism | What's Stored |
|-------------------|-----------|---------------|
| **Cross-session** | `01_DISCOVERY.md` | Repo map + symbol index (survives between conversations) |
| **Intra-session** | Claude Code context window | Dependency graph + context pack + reranking results |

**Invalidation:**
- Cross-session: Re-run `/discover` or `/repo-map` when files change significantly
- Intra-session: N/A — context window is ephemeral by design

## Data Model

### Symbol Index Record

```
type: string    # func | class | iface | type | const | export | enum | trait | struct
name: string    # Symbol name (e.g., "calculateTotal")
file: string    # Relative file path (e.g., "src/billing/calculator.ts")
line: number    # Line number of definition
```

**Storage:** Compact text format in `01_DISCOVERY.md`, one record per line: `{type}:{name}:{file}:{line}`

### Dependency Edge

```
source: string  # File path of the importing file
target: string  # File path of the imported module
kind: string    # "imports" | "tested-by" | "exports-to"
```

**Storage:** In-context only. The LLM builds a mental adjacency list.

### Context Pack

```
files: [
  { path: string, reason: string, score: number }
]
max_files: 8
```

**Storage:** In-context only. Used to guide which files the LLM reads in Steps 1-2.

## API Design

N/A — this is a prompt-engineering system. No HTTP APIs or RPC endpoints.

**Tool interfaces consumed (existing, unchanged):**

| Tool | Used By | Purpose |
|------|---------|---------|
| `Glob(pattern)` | Symbol index, dependency graph | Find source files |
| `Grep(pattern, path)` | Symbol index, dependency graph, import tracing | Extract symbols, find imports |
| `Read(file_path)` | Context pack builder | Read assembled context files |
| `search_code(path, query)` | Level 2 retrieval (optional) | Semantic search for candidates |

## State Management

```
/discover
  │
  ├── Generates: Repo Map + Symbol Index → writes to 01_DISCOVERY.md
  │                                         (persistent cross-session)
  ▼
/research (researching-code skill)
  │
  ├── Step 0a: Reads 01_DISCOVERY.md → loads map + symbol index
  ├── Step 0b: Builds dependency graph → in-context (ephemeral)
  ├── Step 0c: Level 2 search → raw results (ephemeral)
  ├── Step 0d: Reranks results → reranked list (ephemeral)
  ├── Step 0e: Builds context pack → assembled file list (ephemeral)
  ├── Steps 1-2: Uses context pack for deep analysis
  │
  ▼
  Writes: 02_CODE_RESEARCH.md (persistent)
```

**State transitions:** All intermediate state (dependency graph, reranked list, context pack) is ephemeral — held in the LLM's context window during the research phase only. Only the final output (RESEARCH.md) and the structural overview (symbol index in DISCOVERY.md) are persisted.

## Error Handling Strategy

**Core principle: Every component degrades gracefully to existing behavior.**

| Component | Error Condition | Degradation |
|-----------|----------------|-------------|
| Symbol index | No `01_DISCOVERY.md` exists | Generate quick repo map on-the-fly (existing fallback) |
| Symbol index | Index section missing/empty | Proceed with repo map only (existing Level 1) |
| Dependency graph | Import patterns don't match (unusual language) | Skip graph; proceed to Level 2 search directly |
| Dependency graph | Grep returns too many results (noisy) | Take top 20 import matches; note imprecision |
| Reranking | ≤ 5 candidates | Skip reranking; use all candidates |
| Reranking | No dependency graph available | Score using keyword overlap + file-type only (2/3 factors) |
| Context pack | Expansion exceeds 8 files | Drop test files first, then lowest-scored imports |
| Context pack | No dependency graph for expansion | Use seed files only (no expansion) |
| MCP search_code | Not configured / error | Fall back to Grep (existing fallback) |

**No user-facing error messages needed.** All degradation is silent — the LLM adjusts its approach without notifying the user. The quality check at the end of the research phase verifies whether the intelligence components were used.

## Performance Considerations

**Token efficiency (the primary performance metric for a prompt system):**

| Component | Token Cost | Optimization |
|-----------|-----------|-------------|
| Repo map | ≤ 2K tokens | Progressive truncation tiers by repo size |
| Symbol index | ≤ 1K tokens | Same tiers; compact `type:name:file:line` format |
| Dependency graph | 0 tokens (in-context reasoning) | Not written to file; LLM reasons about it |
| Reranking | 0 tokens (in-context reasoning) | Mental scoring procedure, not output |
| Context pack | 0 tokens (file list) + cost of reading files | Max 8 files; progressive read depth |
| **Total structural overhead** | **≤ 3K tokens** | Down from unbounded Grep + Read in current system |

**Tool call efficiency:**

| Phase | Before (current) | After (with intelligence layer) |
|-------|-------------------|-------------------------------|
| Step 0 | 1-3 Grep calls (broad) + Read discovery | 2-5 Grep calls (targeted) + Read discovery |
| Steps 1-2 | 5-15 Grep + Read calls (exploratory) | 3-8 Read calls (context pack, precise) |
| **Total** | **~15-20 tool calls** | **~10-13 tool calls** |

The intelligence layer adds 1-3 Grep calls upfront (for imports) but eliminates 5-10 exploratory calls later by providing a precise context pack.

## Scalability Notes

| Repo Size | Behavior | Notes |
|-----------|----------|-------|
| **Small** (< 50 files) | Skip dep graph + reranking; use repo map → direct file reading | Full pipeline is overkill |
| **Medium** (50–200 files) | Full pipeline: map + index + graph + rerank + pack | Sweet spot for the design |
| **Large** (200–500 files) | Truncated symbol index; primary dirs only for graph | Progressive degradation |
| **Very large** (> 500 files) | Directory-level map; top 30 symbols; graph for top 10 candidates only | Heavy truncation; relies on MCP search_code for precision |
| **Monorepo** (> 1K files) | Path-scoped: `/repo-map src/auth` → full detail for subtree only | User directs scope |

**At 10x current load (2K files):** The system self-limits via progressive truncation. Repo map drops to directory summaries. Symbol index covers only top 30 files. Dependency graph is built for the 5-10 most relevant candidates only. Context pack stays at 8 files max.

**At 100x current load (20K files):** Same truncation mechanisms. The repo map becomes a high-level directory summary (~10 lines). The symbol index is essentially empty (too many to fit). The system relies primarily on MCP `search_code` for Level 2 retrieval, with the dependency graph providing expansion from seed results. This is acceptable — very large repos require semantic search, not structural overview.
